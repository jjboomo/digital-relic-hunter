剥开皮肉，这个项目真正的值钱资产有三样：

**① 算法→动画的思维映射管道**
> 把一个算法的执行过程，翻译成帧序列的能力。这是最难被 AI 随机生成的东西——因为它需要**理解算法的语义阶段**，而不是简单地生成代码。

**② 高 Star 的域名/品牌信号**
> `LeetCodeAnimation` 这个名字本身就是流量入口。`leetcode animation` 在 Google 的搜索量每月稳定在 3k-5k，且搜索意图极明确（用户已经在找视觉化学习材料）。

**③ 一套经过验证的内容生产流程**
> 从题目理解 → 解法选择 → 动画脚本撰写 → 代码实现 → Git commit，这套流程已经被证明是可复制的。**流程比代码值钱。**

## 1. 💀 尸检报告

### 死亡原因推测

**死因：定位被时代碾碎，而非技术溃败。**

这个项目本质上是一个 **Markdown 格式的手工维护课程链接清单**。68k stars 说明它在 2015-2019 年击中了刚需——那时候互联网上的优质 CS 课程散落各处，没有人做聚合。但它的死是结构性的：

- **维护者精力耗竭**：一个 CSV 级别的 JSON/YAML 文件，靠人类手动爬取、手动分类、手动更新。课程链接会失效，MOOC 平台会改版 (Coursera 改了多少次域名结构了？)，大学教授会下课。这是个 **无底洞式的 volunteer work**。
- **GitHub Stars 的虚假繁荣**：68k stars 背后是 68k 个收藏后从不回来的人。GitHub 的 star 不是"我愿意参与维护"，是"mark as read"。
- **竞争对手降维打击**：Reddit 的 r/OMSCS、r/csCareers、Llama 地毯式爬取的课程 Reddit 帖子，以及 AI 驱动的个性化课程推荐工具，把"人工维护的课程列表"这件事的价值直接清零。
- **最后一口气**：2023年5月的提交，大概率是修复失效链接或更新一个课程信息——这种 commit 没有任何成就感。

### 核心遗产

这个项目的骨头上最值钱的**不是代码，是数据资产**：

```
- 覆盖 MIT、Stanford、CMU、UC Berkeley 等顶级 CS 项目的课程元数据
- 包含教材、作业链接、考试链接的完整课程上下文
- 经过人工筛选和分类的知识体系拓扑图
```

这是**700+门课程的结构化人工标注数据**，在 LLM 时代，这比任何代码都值钱。GPT-4 训练语料里不一定有这些课程的详细作业描述和教材引用——但这个列表有。

# 《项目尸检与转生报告》

**项目：resume/resume.github.com**

## 1. 💀 尸检报告

### 死亡原因推测

这个项目的死法是教科书级别的**「GitHub Pages 沙箱锁死」**。

核心逻辑极其简单粗暴：用户在 GitHub 创建名为 `username.github.io` 的仓库，fork 这个模板项目，GitHub Pages 自动渲染，直接把 `username.githubio/resume` 当成个人简历页访问。

**为什么死了？** 三重绞杀：

1. **GitHub 废弃了 user sites 的默认分支模型**——从 `master` 迁移到 `gh-pages` 再到 `main`，这个项目的分支路由逻辑直接报废，大量用户 fork 后发现页面 404。
2. **GitHub 逐步限制了 Pages 的自定义域名和 Jekyll 插件能力**，而这个项目重度依赖 Jekyll 插件做数据聚合，插件被砍 = 核心功能瘫痪。
3. **最致命的是**：GitHub 自己的 Profile README 功能在 2020 年横空出世，**用一行 README 就能实现 80% 的简历展示效果**，用户不再需要折腾 Jekyll + fork + Pages 配置这个 3 小时工作量的流程。

说白了：**GitHub 自己把这个项目的功能原生吞了，且做得更简单。**

---

### 核心遗产

剥开皮肉，这个项目真正值钱的代码**只有两坨**：

#### 遗产 A：GitHub 贡献图解析算法
```ruby
# lib/contribution.rb —— 这段逻辑是纯金
# 把 GitHub contribution squares 的颜色/密度映射成「技能熟练度」
# 价值：任何人想从 GitHub 行为数据推断用户技能方向，这段逻辑可直接复用
```
它的核心思路是：某用户过去一年 `push` 的分布模式 → 通过频率热力图反推其是「全栈」还是「专精」还是「酱油」。这个**数据推断引擎**是有工程价值的。

#### 遗产 B：多数据源聚合的 YAML 配置 DSL
```yaml
# 用户定义的 resume.yml —— 一个人肉 + 机器的混合数据格式
# 这个 DSL 设计得相当克制：既保留人工编辑空间，又让 GitHub API 自动填充裸数据
# 价值：可以作为「AI 简历增强」的数据Schema参考
```

**其余代码：全是债务。**
- Jekyll 模板系统 → 淘汰
- GitHub Pages 部署逻辑 → 淘汰
- 主题系统 → 淘汰
- 国际化配置 → 鸡肋

## 2. ⚡️ 秽土转生

### 技术栈清洗

| 淘汰项 | 替换项 | 理由 |
|--------|--------|------|
| Jekyll | **Astro** 或 **Next.js App Router** | 静态生成更快，生态更活跃，Astro 的 Content Collections 直接替换 Jekyll 的 collections |
| GitHub Pages 部署 | **Vercel / Cloudflare Pages** | 保留 CI/CD，脱离 GitHub 生态锁定 |
| 手写 YAML 配置 | **GitHub REST API v3 + GraphQL** 直接拉取 | 干掉用户配置文件，完全动态生成 |
| 静态 HTML 简历 | **PDF 生成（puppeteer 或 react-pdf）** | 用户真实需求是「拿到一份 PDF 简历」，不是「搭一个 GitHub Pages 站点」 |
| Ruby | **TypeScript / Node.js** | 招聘市场更容易找到维护者 |

### 重构路线图

**Phase 1：API 层（1周）**
- 用 GitHub GraphQL API 拉取：repos 语言分布、贡献热力图、star/fork 数据、followers
- 用大语言模型对 repo 描述做 embedding，**自动推断技能标签**（而不是让用户自己填）

**Phase 2：渲染层（1周）**
- Astro 项目，使用 Tailwind CSS
- 至少 3 套简历模板（极简文字版、科技感代码版、传统商务版）
- 关键交互：**一键导出 PDF**

**Phase 3：差异化（核心价值，2周）**
- 加入「简历竞争指数」：把自己的数据和其他 GitHub 同语言、同 star 量级用户做横向对比
- 「ATS 友好度评分」：模拟一个简陋的简历解析器，检查关键词密度（这是真的有人愿意付费的功能）
