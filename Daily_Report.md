# 📦 猎物：[prakhar1989/awesome-courses](https://github.com/prakhar1989/awesome-courses)

## 1. 💀 尸检报告

### 死亡原因推测

**死因：价值主张被 LLM 直接碾碎。**

这不是一个"项目停更"的问题，这是一个**内容模式被时代淘汰**的问题。

2015-2020 年，awesome-list 模式之所以能积累 68k star，本质上是因为：
- 互联网上的 CS 课程资源极度分散
- 没有一个中心化的发现机制
- GitHub stars 本身就是一种信任背书

**但 2023 年之后，ChatGPT/Gemini/Claude 可以实时回答：**
> "我想入门 ML，但数学基础一般，应该按什么顺序学哪些大学的公开课？"

这个查询的答案质量，**在任何维度上都碾压一个静态 Markdown 文件**。

而 maintainer prakhar1989 最后一次提交是 2023-05-04——这恰好是 AI 教育工具开始井喷的时间节点。**他不是没能力继续维护，是维护的 ROI 归零了。**

---

### 核心遗产

扒开皮肉，这个仓库最值钱的**不是那些链接**，而是：

| 遗产 | 价值描述 |
|------|----------|
| **课程分类 Ontology** | 这套按 CS 领域（OS/Networks/PLT/ML...）→ 课程层级（基础/进阶）组织的元数据 schema，本质上是一套手工构建的"CS 知识图谱" |
| **68k 社区验证数据** | 每一个 star 都是一次"我认为这个分类体系有价值"的无声投票——这是训练课程推荐模型的高质量信号 |
| **细分领域覆盖广度** | 从编译器到分布式系统，从数据库到安全——这套目录结构覆盖了完整的 CS 知识版图 |
| **跨平台权威链接** | Yale CS、MIT OCW、Stanford、CMU... 这些大学的公开课资源聚合本身具有目录垄断价值 |

**一句话总结**：这个 repo 是一个用 GitHub 实现的、靠社区投票排序的、自下而上的 CS 教育知识图谱原型。代码价值趋近于零，**结构化数据价值极高**。

---

## 2. ⚡️ 秽土转生

### 技术栈清洗

| 淘汰项 | 替换项 | 淘汰理由 |
|--------|--------|----------|
| `README.md` 静态文档 | **动态数据库 + RAG** | 静态文档无法个性化，无法问答，无法更新 |
| 手动 PR 提交课程 | **AI Agent 自动抓取 + 人类审核** | 手动提交流程慢、低效、容易成为维护者的负担 |
| GitHub stars 作为排序信号 | **多维度评分系统**（注册人数 + 完成率 + 评价分 + 最近更新时间） | Stars 只反映发现，不反映质量 |
| 按"大学"组织课程 | **按"技能树 / 路径"组织** | 用户关心的不是"哪所大学的课"，而是"学完能做什么" |
| 无状态的链接列表 | **带上下文的 AI 对话界面** | 消歧义：同样是"操作系统课"，MIT 6.S081 和清华的侧重点完全不同 |

**推荐技术栈**：

```
前端: Next.js 14 (App Router) + shadcn/ui
AI层: Claude API / GPT-4o (function calling for course recommendation)
后端: Next.js API Routes
数据库: PostgreSQL + pgvector (course embeddings)
CMS层: 迁移到结构化的 DB schema，而非 README.md
状态: Vercel AI SDK (流式输出)
监控: Plausible (隐私友好的 analytics)
```

### 重构成本预估

| 模块 | 工程量 | 说明 |
|------|--------|------|
| 数据迁移（README → DB） | **1-2 人/周** | 写脚本解析现有 Markdown，按现有 ontology 结构入库 |
| Embedding 向量化 | **0.5 人/周** | 每门课程描述、课程大纲、 syllabus 向量化 |
| RAG 推荐引擎 | **3-4 人/周** | Prompt engineering 是大头——需要让 AI 理解"先修/后继"关系 |
| 前端界面 | **2-3 人/周** | 不需要花哨，核心是"路径可视化" |
| 课程更新 pipeline | **2 人/周** | 自动化抓取 + 人工审核流 |
| **总计** | **8-12 人/周** | 小团队 2-3 个月可 MVP |

# 《项目尸检与转生报告》

## 1. 💀 尸检报告

### 死亡原因推测

**直接死因**：GitHub 在 2016 年移除了 Jekyll 插件支持，斩断了这个项目的自动化血脉。

**深层死因**：
- 核心价值建立在 GitHub API 的**免费滥用**之上，当 GitHub 收紧 API Rate Limit 时，这个项目的竞争力归零
- 用户体验需要手动编辑 YAML，对于 2018 年后的用户来说这已经是“史前操作”
- 致命问题：**GitHub 本身就是简历**，用户为什么需要另一个简历？

**根本原因**：这个项目是个**寄生型工具**，宿主（GitHub）一旦收紧生态，立刻失血。

---

### 核心遗产

| 遗产模块 | 价值评估 | 说明 |

| **GitHub Profile → 简历数据提取算法** | ⭐⭐⭐⭐⭐ | 这是唯一不可替代的东西 |
| **简历模板系统** | ⭐⭐ | 设计过时，但逻辑可用 |
| **数据结构化映射**（语言/仓库 → 技能） | ⭐⭐⭐ | 沉淀了真实用户行为数据模式 |

## 3. 💰 变现杠杆

### 黑客级变现路径

**核心策略**：把简历变成 **GitHub Contribution 的付费增值层**

具体路径：
1. **猎头/HR API 订阅**：直接卖给猎头公司，提供「批量下载候选人 GitHub 简历」的接口，绕过 LinkedIn 的信息孤岛
2. **简历 PDF 加密水印追踪**：在生成的 PDF 里植入隐形追踪码，一旦泄露直接溯源（针对那些偷简历的 HR）
3. **LinkedIn 对撞**：用户提供 LinkedIn 主页 → 自动爬取 + 合并 GitHub 数据，生成双渠道简历

**禁止大平台，但必须寄生**：
- 寄生在 Notion、Linear 这些工具的简历导出场景
- 寄生在招聘平台的简历质量焦虑

**定价锚点**：
- 免费：生成基础简历（水印 + 单次导出）
- 付费 $5-15/mo：去除水印 + 历史版本 + 多格式导出
- 企业版：API 访问 + 白标

> **毒舌结论**：这个项目的价值不在于「简历生成器」，而在于**GitHub 数据提取能力**。把数据提取能力包装成「猎头情报工具」比「个人简历生成器」值钱 10 倍。
